## Customer model

### Persona 1 — Maya, the engineering manager mid-cycle

**Today (without this CLI):** Maya runs a 9-person platform team. On any given afternoon she has Linear open in three tabs (her team's current cycle, the active project, the triage inbox), a Slack window, and a Google Doc draft of Friday's stakeholder update. To answer "are we going to land the API migration this cycle?" she eyeballs the cycle board, mentally counts open issues, and guesses. To answer "who is overloaded?" she scrolls each engineer's assigned list. She cannot answer "show me everything that slipped from last cycle into this cycle, grouped by why" without writing a saved view, and she cannot compare last cycle's completion rate to the previous three cycles without exporting CSVs.

**Weekly ritual:** Friday afternoon she writes a stakeholder update: cycle progress %, what shipped, what slipped, projected project landing dates, and any blocked work. She also runs Monday morning sprint planning where she rebalances assignments based on who closed how much last cycle.

**Frustration:** Linear gives her the data but not the shape. Burndown exists in the UI for cycles but not for projects-with-arbitrary-issue-bags. Cycle-over-cycle comparison requires a spreadsheet. "Who is overloaded right now" requires N tabs and arithmetic.

### Persona 2 — Devon, the staff engineer in the middle of a long-running project

**Today (without this CLI):** Devon owns a 40-issue project that spans two teams and three cycles. They start the day on `linear.app/issues` filtered to "assignee:me, state:!Done", flip to the project page to see what's blocked, then to the triage queue to claim incoming bugs. They keep a terminal open with a feature branch checked out. They keep a TODO.md in the repo with "things I told someone I'd look at."

**Weekly ritual:** Every morning: sweep their assigned list, update statuses on yesterday's work, check what's blocked-by-me to unblock teammates, pick today's focus issue and `git checkout -b` from it. Every Friday: a personal mini-retro where they look at what they actually shipped versus what was on Monday's list.

**Frustration:** Context switching between Linear and the terminal. The "what's blocked because of me" question is buried — Linear shows blocking relations on individual issues but not as a personal queue. And once they create test issues to debug something or experiment with the workflow, those tickets pollute the project and they forget to clean them up.

### Persona 3 — Priya, the product manager doing weekly portfolio review

**Today (without this CLI):** Priya owns three initiatives that span seven projects across four teams. To prep her Tuesday portfolio review she opens each initiative, then each project under it, eyeballs progress bars, copies milestone dates into a doc, and pings the project lead in Slack when something looks off. She keeps a Notion page of "projects to watch" with hand-typed status notes.

**Weekly ritual:** Tuesday portfolio review with leadership: per-initiative health, milestone slippage, projected completion dates, anomalies. Wednesday triage of incoming feature requests across her domain.

**Frustration:** Linear shows project-level progress but not the rolled-up initiative view she needs, and projected completion dates are static (set by whoever made the project) rather than computed from current velocity. She cannot answer "which milestone in my portfolio is most at risk this month?" in one place.

### Persona 4 — Sam, the agent operator using the MCP surface

**Today (without this CLI):** Sam uses Claude Code daily and has bolted Linear onto it via a third-party MCP. The agent can call `getIssue` and `createIssue` but cannot do "for every issue assigned to me that's been idle 7+ days, draft a status comment and ping the assignee" without burning thousands of tokens listing endpoints first. Multi-step rituals (triage, sprint planning) require Sam to script them turn-by-turn.

**Weekly ritual:** Every weekday morning Sam asks the agent "what should I work on today?" and expects a ranked, joined view (priority + cycle deadline + blocked-state) without naming the underlying joins. On Fridays Sam asks the agent to "clean up anything you created this week that's still in test state."

**Frustration:** The endpoint-mirror MCP surface is too thin for ritual workflows and too token-heavy at 60+ tools. The agent loses the plot mid-ritual and asks Sam to confirm trivial subqueries. There is no contract for "tickets the agent itself created" so cleanup is impossible.

### Killed candidates

| Feature | Kill reason | Closest surviving sibling |
|---------|-------------|---------------------------|
| `standup` | Reframed mechanical version is a thin formatter over `today`+`slipped`+`velocity`; users can pipe those to their LLM of choice. Saves no work that warrants its own command. | `today`, `slipped` |
| `workload` | Sibling overlap with `bottleneck` — already exposes per-assignee load and adds blocked counts. Two commands for the same question. | `bottleneck` |
| `at-risk` | Wrapper over `projects burndown` + milestone target join; the value is in burndown itself. | `projects burndown`, `initiatives health` |
| `mine` | Composite of `today` + `blocking`; `today` already ranks. | `today`, `blocking` |
| `pr-ready` | Thin filter wrapper; absorbed `issues list --state="In Review" --assignee=@me` covers it. | absorbed #1 |
| MCP intents (`triage_inbox`, `daily_standup`) | Not novel CLI features — they belong in the spec's `mcp.intents` block per the v3 enrichment plan. | spec MCP enrichment |
